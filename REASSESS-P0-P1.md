## Findings (4 critical/high)

### [CRITICAL] The actor thread can never terminate: run() owns self_tx for its whole lifetime, so the command channel never closes — the documented "actor lives until every Core handle is dropped" is impossible. This leaks a thread + a writable SqliteStore handle per spawn, and (worse) lets a leaked worker double-apply and lets two live writable actors share one db.
- **Evidence:** src/actor.rs:43-49 run() takes self_tx: Sender<Command> and holds it on its stack for the entire function; src/actor.rs:62 `while let Ok(cmd) = rx.recv()` only exits when ALL senders drop, but self_tx is itself a live sender, so recv() blocks forever even after every Core is dropped; src/actor.rs:310-314 each worker also clones tx and outlives Core. lib.rs:75/95 claim the actor "lives until every Core handle is dropped" and grep confirms there is no Drop on Core and no Command::Shutdown. tests/p1_reentrant.rs:183-185 explicitly assume drop(core_a) "stops its actor" and that the leaked unit-1 worker later sends "into a closed channel — never double-applying". That assumption is false: actor A stays blocked in recv(), still owns its writable store; Core B (p1_reentrant.rs:188-191) opens the SAME db (a second live writable actor — single-writer violated); when release_tx drops at end of test the leaked worker's send IS received by the still-alive actor A and re-runs apply_step_result on the already-completed unit (double-apply / store corruption). The test passes only because this happens during teardown after assertions.
- **Fix:** Give the actor a real shutdown path so the loop ends when external handles are gone (and in-flight workers have drained), instead of relying on channel closure that self_tx defeats. E.g. track live Core handles with an Arc and on the last Drop send Command::Shutdown; on Shutdown break once in_flight is empty (set a draining flag otherwise). Independently, take an OS-level exclusive file lock on the db in the actor so a second writable actor on the same path fails fast rather than silently double-writing, and make worker write-back robust to a gone actor (it already uses `let _ =`, but the actor must actually be gone).
- *(lens: Concurrency correctness of the actor)*

### [HIGH] apply_step_result has no idempotency/cursor guard — it blindly applies units[output.unit_ix] without checking it matches the session cursor, so any duplicate or stale ApplyStepResult re-applies an already-finished unit (duplicate conformance claims, extra tick_workflow, cursor over-advance). This is the corruption mechanism behind the critical finding and a defense the per-actor in_flight HashSet cannot provide across actors/processes.
- **Evidence:** src/actor.rs:229-256 apply_step_result reads session + units and calls pipeline::apply_and_finish_unit on units[output.unit_ix] unconditionally; it never compares output.unit_ix to session.unit_ix nor checks unit.status. The in_flight guard (actor.rs:60, local HashSet) only blocks duplicate LaunchRun/ResumeRun within one actor; it does nothing for a stale ApplyStepResult or a second actor (workflow.rs StepInput.attempt exists for event-id dedup but is unused here).
- **Fix:** At the top of apply_step_result, fetch the session and early-return Ok(()) if output.unit_ix != session.unit_ix (or if units[output.unit_ix].status is already Done) — treat it as a stale/duplicate worker result and ignore it. This makes step application idempotent regardless of where the duplicate came from.
- *(lens: Concurrency correctness of the actor)*

### [MEDIUM] in_flight removal on the ApplyStepResult path is derived by re-reading store status (run_finished) rather than from the control-flow outcome apply_step_result already knows. A silently-ignored put_node failure in finalize_run then wedges the run as permanently RunBusy (can never resume or relaunch), and the removal logic is inconsistent with the LaunchRun/ResumeRun paths that remove explicitly.
- **Evidence:** src/actor.rs:209-222: after apply_step_result returns Ok, the handler calls run_finished(&store,&run_id) (actor.rs:333-338, status==Completed) to decide whether to drop from in_flight; src/actor.rs:319-330 finalize_run does `let _ = put_node(...)` so a write failure leaves status != Completed → run_finished is false → in_flight never cleared → every future LaunchRun/ResumeRun returns RunBusy forever. Contrast actor.rs:155-157 and 199-206 which remove explicitly.
- **Fix:** Have apply_step_result return an enum {DispatchedNext, Finished} and remove from in_flight based on that value, not a store re-read. Propagate/surface the finalize_run put_node error instead of swallowing it so a failed finalize doesn't masquerade as a still-running (busy) state.
- *(lens: Concurrency correctness of the actor)*

### [MEDIUM] resume_run has no persisted lease/in-flight marker, so the in_flight guard is purely per-actor. Two actors on the same db (the leaked-actor case above, or a real crash-recovery race where the old executor isn't actually dead) will both dispatch the same unit with no cross-actor mutual exclusion — the exact double-dispatch the design says it prevents.
- **Evidence:** src/actor.rs:165-208 ResumeRun checks only the local in_flight HashSet (actor.rs:60) before re-dispatching from session.unit_ix; nothing in the store records that a unit is currently executing. The crate is explicitly multi-consumer/out-of-process (gate-hook runs in a separate process per command.rs:31-35), so "only one actor ever touches this db" is not guaranteed at runtime.
- **Fix:** Persist an execution lease on the session node (e.g. executing-since timestamp / owner token) that resume_run checks and claims atomically, expiring only after a TTL, so a second actor refuses to re-dispatch a unit another actor is actively running. Combined with the high-severity cursor check this makes resume safe even when an old executor lingers.

### [HIGH] The gate-hook is NOT a read-only / zero-write store user: open_store() opens the SQLite file READ-WRITE and writes on every open (PRAGMA journal_mode=WAL + synchronous + auto_vacuum, execute_batch(SCHEMA) DDL, and migrate_schema ALTERs on legacy DBs). There is also no busy_timeout. So the per-tool-call hook subprocess is a second OS-process that can take the SQLite write lock at open, and under contention with the actor's write transaction it gets immediate SQLITE_BUSY (busy_timeout=0) and fails CLOSED -> a spurious governance DENY that aborts the user's tool-call.
- **Evidence:** gate_hook.rs:74-81 calls open_store(db) and the doc at gate_hook.rs:8-12/56 claims 'ZERO store writes' / 'opened read-only in effect'. But wicked-estate-store-0.12.0/src/sqlite.rs:462-473 SqliteStore::open runs `PRAGMA journal_mode=WAL; ...; auto_vacuum=INCREMENTAL`, then execute_batch(SCHEMA) (CREATE TABLE...), then migrate_schema (ALTER TABLE on legacy DBs). sqlite.rs has no busy_timeout/busy_handler and no READ_ONLY OpenFlags (grep empty). On a fresh/legacy DB the hook is literally the schema writer; on a steady-state DB the DDL is a no-op but WAL/pragma + DDL still open a write path with zero busy wait.
- **Fix:** Give the hook a genuinely read-only path: open with SQLITE_OPEN_READ_ONLY (skip WAL pragma/DDL/migrate -- the actor owns schema creation), or at minimum add a busy_timeout (e.g. 5s) so a concurrent actor write does not turn into a spurious DENY, and correct the module doc which currently overstates the invariant as 'ZERO store writes / read-only in effect'. The app-level single-writer (actor) guarantee still holds; this is about the hook open path not being the pure reader it is documented to be.
- *(lens: SINGLE-WRITER + GOVERNANCE INVARIANT preservation)*

# P0+P1 Reassessment → Revised P2-P7 Roadmap

## 1. Verdict on P0+P1

**P0 is sound and shippable as a foundation. P1's architecture is sound but the actor lifecycle has a genuine correctness hole that must be closed before P2 builds on it.** The off-thread split, the `plan_and_distribute`/`apply_and_finish_unit` extraction (pipeline.rs:117/205), the injectable `StepRunner`/`Dispatcher` seams (lib.rs:87), and the resume-from-cursor mechanism (actor.rs:185-208 reading `session.unit_ix`) are the right shapes and directly enable P2-P7. But the engine as written cannot terminate an actor, and step application is not idempotent — both of which P2 (retry, cancel, confirm — all of which re-enter `step`) will turn from latent into active corruption.

### MUST-FIX-NOW (before any P2 code)

- **Actor lifecycle (the critical).** `actor::run` takes `self_tx: Sender<Command>` (actor.rs:46) and holds it on its own stack for the whole loop; workers also clone it (actor.rs:310). The recv loop (actor.rs:62) only exits when *all* senders drop, so it never exits — confirmed: no `Command::Shutdown`, no `impl Drop` anywhere in `src/`. `drop(core_a)` in p1_reentrant.rs:185 does **not** stop actor A; the doc claims at lib.rs:74 ("lives until every Core handle is dropped") and the test comment at p1_reentrant.rs:183-184 are false. Consequence: every spawn leaks a thread + a writable `SqliteStore`, and the test's core_b (p1_reentrant.rs:188) opens the **same db while actor A is still alive and writable** — a live single-writer violation that the suite only survives because it happens after assertions. **Fix now:** add `Command::Shutdown` + an `Arc` handle-count (or `Weak` self_tx) so the loop ends when the last `Core` drops and `in_flight` is drained; do NOT clone `self_tx` into the actor's own stack as a strong sender. This is foundational because P2's `CancelRun`/`PauseRun`/`ConfirmGate` multiply the number of live actors and re-entries.

- **Idempotency/cursor guard in `apply_step_result` (the high that is the corruption mechanism).** actor.rs:229 applies `units[output.unit_ix]` unconditionally — no check that `output.unit_ix == session.unit_ix` or that `units[ix].status` is not already `Done`. The per-actor `in_flight` HashSet (actor.rs:60) cannot defend against a stale/duplicate `ApplyStepResult` (which is exactly what the leaked worker in the test produces). **Fix now:** at the top of `apply_step_result`, reload session and early-return `Ok(())` if `output.unit_ix != session.unit_ix` or the unit is already `Done`. This is ~5 lines and makes step application idempotent regardless of source — it is the cheapest, highest-leverage fix in the set and it directly hardens P2's re-entrant confirm/retry paths. The unused `StepInput.attempt` (workflow.rs:33) should be folded into this check now since P2 needs attempt-scoped dedup anyway.

### SAFE-TO-DEFER (with reasoning)

- **`in_flight` removal via store re-read (`run_finished`, actor.rs:219-222) → make it control-flow-driven (medium).** Real wedge risk: `finalize_run` swallows the `put_node` error (actor.rs:322 `let _ =`), so a failed finalize leaves status ≠ Completed → `run_finished` false → `in_flight` never cleared → permanent `RunBusy`. But it only bites on a store-write failure, which the stub never hits. Defer to the start of P2 and fix together with the lifecycle work (have `apply_step_result` return `{DispatchedNext, Finished}` and propagate the finalize error) — it's the same code region.

- **Persisted execution lease / cross-actor mutual exclusion (medium).** Today only one actor *should* touch a db, and once the lifecycle fix lands the leaked-second-actor scenario disappears. The cursor guard above makes a double-dispatch idempotent even without a lease. A real OS-level file lock + persisted lease is genuinely needed, but it belongs with **P3** (worktree/repo lifecycle + crash recovery, where "is the old executor actually dead?" is the native question), not P2.

- **gate-hook is not truly read-only (high, but not now).** gate_hook.rs:75 `open_store` opens RW and writes WAL pragma/DDL/migrate with no `busy_timeout`. The module doc ("ZERO store writes / read-only in effect", gate_hook.rs:8-12/56) overstates the invariant. Today the hook is invoked directly in-process by the P0 test, never concurrently with an actor write, so no spurious DENY yet. This becomes a **P4b precondition** (when the hook is a real per-tool-call subprocess racing actor writes): open with `SQLITE_OPEN_READ_ONLY` (skip pragma/DDL/migrate — the actor owns schema) or at minimum set `busy_timeout`. Fix the doc now (one comment), fix the open path in P4b.

- **Both deferrals (advance()-sole-opener, event seq).** See §2. Seq is genuinely safe to defer to P7 (single emit point verified: actor.rs:351 `emit` only called on actor thread; workers at actor.rs:311-313 only `send`, never emit). The phase-opener deferral is safe *for now* but is a hard P4a precondition (see §2).

## 2. What we learned / what changed

- **The actor protocol is more subtle than §3.3 assumed; the `self_tx` write-back pattern needs a lifecycle co-design.** The design specified the `Sender<Command>` clone for write-back (§3.3, critique-resolution row "Actor needs a Sender<Command> clone") but never reconciled it with "actor lives until every Core handle is dropped" (§1). These two are contradictory: a strong self-sender defeats channel-close termination. **Delta:** add an explicit `Shutdown`/handle-count lifecycle to the P1 deliverable list (it was implicit and got missed). This also retroactively makes `CancelRun`/`PauseRun` (P2) implementable — cancellation needs the same actor-reachable handle plumbing.

- **The `plan_and_distribute` / `apply_and_finish_unit` extraction (pipeline.rs:117/205) is a real asset and changes later phases for the better.** Both drivers (sync `run_session` at pipeline.rs:38 and the actor's `apply_step_result` at actor.rs:247) provably call the same shared steps, so there is no inter-driver drift. This means **P4a only has to replace the work-*production* seam (`StepRunner::run_unit`, workflow.rs:55), not the gate/cursor/evidence application** — the apply half is already unified and tested. P4a shrinks accordingly.

- **The leaked-worker pattern implies a worker-handle / cancellation need is needed EARLIER than P2's "nice to have."** `dispatch_unit` (actor.rs:311) spawns a detached `std::thread::spawn` with no join handle, no cancellation token, and no way for the actor to know a worker is orphaned. The design's §2.6 `CancellationToken` + child-process handle was scoped to P2/P4a, but the *handle-tracking* half (so the actor can drop/ignore a worker whose run was superseded) is now a prerequisite for the lifecycle fix. **Delta:** pull worker-handle tracking (a `HashMap<run_id, WorkerHandle>` alongside `in_flight`) into the P2 lifecycle slice; defer only the real subprocess kill signal to P4a.

- **Dual-cursor drift on denial is real data drift (medium), and it changes when the advance() refactor must land.** `apply_step_result` advances `session.unit_ix` and `finalize_run` marks the session `Completed` regardless of `outcome.approved` (actor.rs:259, 321), while `tick_workflow(approved=false)` (pipeline.rs:235) latches `WorkflowStatus::Failed` and freezes `current_index`. End state of a denied run: workflow node `Failed`/frozen, session `Completed`/`unit_ix=len`. Latent today (nothing reads `current_index` for control flow), but it will mislead the real backend. **Delta:** the §2.1 unification (one cursor read from `Workflow.current_index`, delete `tick_workflow`, make execute phase-pure so `advance()` is sole opener) is **a hard P4a precondition**, not optional polish — confirmed deferral (1) is entirely unimplemented (execute.rs:70 `Phase::open` + pipeline.rs:235 `tick_workflow` both live). The good news: there is **no double-open today** because `advance()` is unused and hook phase ids (`wf-x:exec`) never collide with unit ids (`wf-x:unit-N`). So this can stay deferred through P2/P3 and be done as the first task of P4a. Update the §8 P1 deliverable status from "done" to "deferred to P4a" for this item.

- **Run-level outcome on denial is unspecified and arguably wrong.** A `Rejected` unit (pipeline.rs:241) still yields a `Completed` session with no distinguishing status. P2 must decide the run-level contract (terminal `Failed`, or `Completed-with-rejections`) because P2's whole point is gates that *halt*.

## 3. Revised next slice

**The single highest-value next thing is a small "P1.5 hardening" slice (lifecycle + idempotency), then P2 as planned.** Do NOT start P2 features on top of an actor that cannot terminate and a step-apply that double-applies — P2's confirm/retry/cancel are all re-entrant `step` calls and would compound both bugs. P1.5 is ~half a session: `Command::Shutdown` + handle-count lifecycle, the `apply_step_result` cursor/status guard, control-flow-driven `in_flight` removal, surfaced `finalize_run` error, worker-handle tracking, and the `gate_hook` doc correction. Add the three cheap missing P1 tests (resume-of-Completed, empty-problem, two concurrent runs) and a **blocking-dispatcher** test (the §3.3 off-thread claim is currently only proven for the *runner*, not the *distribute* phase — distribute still runs synchronously on the actor at actor.rs:125 with the real subprocess dispatcher).

**Then P2 remains the right next phase** (typed stages + clearable human-confirm gates + retry + cancel). Rationale: it is the first phase that delivers user-visible interactivity, it exercises the safety-critical clearable-veto invariant (the one thing the engine exists to guarantee), and it forces the run-level deny contract decision that P1 left open. P2 is also where worker-handle/cancellation lands, which the leaked-worker finding showed is overdue.

**Resequencing / parallelization:**

- **Strictly sequential:** P1.5 → P2 → P4a. P4a depends on P2's typed stages AND on the deferred advance()-sole-opener refactor (§2). Do the cursor-unification refactor as P4a's first task.
- **Independent of P2, parallelizable now:** **P3 (repo registry + worktree)** is purely additive (new `RepoEntry` node-kind, git plumbing, lifecycle) and touches none of the actor protocol — it can be built concurrently with P2 by a separate track. The persisted-lease/file-lock work (deferred medium findings) naturally folds into P3.
- **Independent of each other, but both gated on P4a:** **P5 (council adversarial)** and **P6 (functional testing)** are both stage *backends* that plug into the P4a `StepRunner` seam. Once P4a exists they can be built in parallel. P5 needs no worktree; P6 needs P3's worktree (`process.cwd()` for wicked-testing). So: P5 ∥ (P3→P6).
- **P4b (gate-hook re-home)** is independent of P4a's execute backend and can run in parallel with it, but must precede any real `claude -p` run; it's where the gate_hook read-only fix lands.
- **P7 (egui)** is last and depends on the `CoreEvent` surface being stable (so add the seq stamping here).

Revised critical path: **P1.5 → P2 → P4a → {P5 ∥ P6}**, with **P3 ∥ P2**, **P4b ∥ P4a**, **P7 last**.

## 4. Streamlining opportunities

- **Drop the legacy `Command::Launch` / sync `run_session` path once P2 lands.** The actor carries two execution shapes (Launch straight-through at actor.rs:78, and LaunchRun/ResumeRun). The shared extraction means the only thing the sync path still provides is the operator-CLI convenience. After P2, `run_session` can be reimplemented as "launch_run + block until SessionCompleted" over the interactive engine, deleting the dual-shape branch and the `Command::Launch` arm. Fewer code paths to keep drift-free.

- **Merge the two deferred mediums into P2's lifecycle commit** (control-flow `in_flight` removal + surfaced finalize error). They live in the same actor.rs:209-222 / 319-330 region the lifecycle fix touches; doing them together avoids a second pass.

- **Collapse `StepOutput` and the design's `StepResult` now, cheaply.** `StepOutput` (workflow.rs:43) lacks the `status: StepStatus {Ok|Failed|Cancelled|Interrupted}` the design specifies (§3.4). Add the discriminant in P1.5/P2 (wire only `Ok|Failed` initially) so `apply_step_result` can branch on worker failure instead of encoding errors in the output `String`. Cheaper now (one runner, one apply site) than after P4a/P5/P6 add three more runners.

- **Per-phase gate aggregation: pick a rule and document it (medium gate-hook finding).** `apply_hook_decisions` builds one `phase_id = {workflow_id}:{claim.phase}` per phase (gate_hook.rs:174), so it's first-decision-wins; an Allow-then-Deny leaves the phase `Approved` even though a Deny was conformed. For P2's release/merge gates that read phase *status* this is wrong. Cheapest fix: in the drain, scan a phase's claims and force `Rejected` if any Deny exists (deny-wins), OR give each tool-call a sub-phase id. Decide in P2 (where gate semantics are the topic), not later.

- **Don't build `AssignClis`/`ChatWithCli`/`PauseRun` in P2 — keep P2 to the gate/retry/cancel core.** The design's §3.1 lists many commands; the proving value of P2 is the clearable-veto invariant. `ChatWithCli` is a P7 concern (UI), `AssignClis` is a P4a/P5 concern (real multi-CLI dispatch), and `PauseRun` can fold into the `CancelRun` worker-handle machinery. Scoping P2 tighter keeps the safety-critical reducer change isolated and reviewable.

- **The `next_cli_in_roster` wrap-around (pipeline.rs:276) is a known evaluator=creator hazard** flagged in §2.4.1 — it currently falls back to `"wicked-evaluator"` rather than silently grading itself, which is acceptable, but P5 must replace it with the validated distinct-seat guard. No action before P5; just don't let P4a's multi-CLI work re-introduce reliance on roster position.
